# Historical Data Pipeline: Norwegian Psychiatric Records (1872–1929)

This pipeline digitizes and structures annual admission records from Norwegian psychiatric institutions into a research-ready long-format dataset. It covers the period 1872–1929 across all major asylums, recording patient counts by institution, diagnosis, and sex.

---

## Pipeline Overview

```
01_pdf  →  02_json_raw  →  validation_shiny_app.R  →  03_json  →  combine.R  →  cleaned_long_dataset.csv
(source)   (AI skeleton)    (human data entry)         (verified)   (export)      (final dataset)
```

---

## Directory Structure

| Directory | Purpose |
| :--- | :--- |
| **`01_pdf`** | Original scanned source documents — one PDF per year of the annual asylum report. |
| **`02_json_raw`** | Raw JSON skeletons generated by the AI extraction prompt. Contains the structural dimensions (institutions, diagnoses) but no numeric values. |
| **`03_json`** | Verified, complete JSON files — one per year — with all numeric values and footnote markers entered by hand. This is the source of truth for the final dataset. |

---

## Core Files

| File | Purpose |
| :--- | :--- |
| **`ai_prompt.txt`** | The prompt used to extract table structure (institutions, diagnoses, year) from each source PDF, producing the raw skeleton in `02_json_raw`. |
| **`validation_shiny_app.R`** | Interactive data entry tool. Loads a raw skeleton or a previously saved file, allows manual entry of all numeric values and footnotes, and saves the completed JSON to `03_json`. |
| **`combine.R`** | Final export script. Reads all verified JSONs from `03_json`, standardises institution and diagnosis names, filters aggregate rows, and exports `cleaned_long_dataset.csv`. Includes a built-in audit verifying that each year's total matches the proofed reference counts. |

---

## Data Entry Workflow

### Step 1 — AI Extraction
Feed a year's PDF into the AI prompt. It returns a JSON skeleton in `02_json_raw` containing the table's structure (institutions × diagnoses × sex) with no values filled in.

### Step 2 — Human Validation (Shiny App)
The Shiny app has two import modes:

- **Dimensions Mode:** Load a raw skeleton from `02_json_raw`. The app expands it into a full entry grid (Cartesian product of all dimensions) ready for data entry.
- **Resume Mode:** Load a previously saved `_processed.json` from `03_json` to continue editing or correct errors.

The entry interface has two synchronized grids:
- **Data Grid** — numeric values only.
- **Footnote Grid** — non-numeric markers (e.g. `*`, `1)`, `a`). Cells with footnotes are flagged visually in the Data Grid.

Institution and diagnosis names can be renamed globally — one change propagates instantly across both grids and all saved data.

A single click saves the completed file directly to `03_json` as `YYYY_processed.json`.

### Step 3 — Export
Run `combine.R`. It reads every file in `03_json`, applies name standardisation and aggregate filtering, and writes `cleaned_long_dataset.csv` — a long-format table with columns for year, institution, diagnosis, sex, value, and footnote marker. The script prints a year-by-year audit against verified reference totals before exporting.

---

## Final Dataset

`cleaned_long_dataset.csv` is long-format with one row per (year × institution × diagnosis × sex) cell:

| Column | Description |
| :--- | :--- |
| `year` | Report year (1872–1929) |
| `institution` | Standardised asylum name |
| `patient_status` | Standardised diagnosis |
| `sex` | `Md.` (male) or `Kv.` (female) |
| `value` | Admission count (integer; `NA` if blank in source) |
| `footnote_marker` | Non-numeric marker from source table, if present |
